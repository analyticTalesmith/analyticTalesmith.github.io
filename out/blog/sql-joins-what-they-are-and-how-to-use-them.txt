<p class="">You have multiple tables of related data, and you need to quickly access them both. You <em>could </em>try to use the tables separately, or you might go into Notepad or Excel to join them manually. But how can you speed up the process of joining the data? Well, conveniently, SQL has a set of functions called &#8220;joins&#8221; that can do exactly what they say: join data from multiple tables. So what exactly are joins, how do they work, and what types of joins are available to you?</p>



<!--more-->



<div class="wp-block-stackable-heading stk-block-heading stk-block-heading--v2 stk-block stk-f4bc9fb" id="introduction-to-sql-joins" data-block-id="f4bc9fb"><h2 class="stk-block-heading__text">Introduction to SQL Joins</h2></div>



<div class="wp-block-stackable-text stk-block-text stk-block stk-8e2ad02" data-block-id="8e2ad02"><p class="stk-block-text__text">A SQL join clause looks through two tables and combines similar rows based on the join condition you provide, returning the combined rows in one table for you to use in your queries. This condition will generally tell SQL what specific key (usually a primary key in one table that is present as a foreign key in another table) should be used to match rows across tables. Joins are like a matchmaking process for your tables, and you can use multiple joins to combine more than 2 tables.</p></div>



<div class="wp-block-stackable-text stk-block-text stk-block stk-8bec64d" data-block-id="8bec64d"><p class="stk-block-text__text">It is important to note that joins will return the subset of the cartesian product of the original tables that meet the conditions you specify in your query. That is, given table A with <em>n </em>rows and table B with <em>m </em>rows, a join will return at most <em>n</em>*<em>m </em>rows. The subset will only reach this max length if your join condition is True for<em> all</em> pairs of rows. The dataset returned by your join will contain a row for each instance where the join condition was True (or in the case of outer joins, for each row that could not be matched).</p></div>



<div class="wp-block-stackable-text stk-block-text stk-block stk-cbc7268" data-block-id="cbc7268"><p class="stk-block-text__text">Why does this technically matter? It is entirely possible for the returned dataset from your join to be longer than either original table. Say we have one table of 3 customer IDs and another of 15 purchases. If instead of joining based on customer ID, I accidentally passed my join condition as &#8220;1 = 1&#8221; (i.e., always True), then the resulting dataset will contain 45 rows, one row for each purchase joined with each customer (the exact same dataset you would get if you used a cross join without a where clause).</p></div>



<div class="wp-block-stackable-heading stk-block-heading stk-block-heading--v2 stk-block stk-558a276" id="types-of-joins" data-block-id="558a276"><h2 class="stk-block-heading__text">Types of Joins</h2></div>



<div class="wp-block-stackable-heading stk-block-heading stk-block-heading--v2 stk-block stk-511c764" id="inner-join" data-block-id="511c764"><h3 class="stk-block-heading__text">Inner Join</h3></div>

<p class="has-d-8-dee-9-ff-color has-text-color" id="inner-join">Inner joins return all joined rows where the key in B matches the key in A. Inner joins are the default join type in SQL. You <em>could </em>just use join in your query and the results would be the same as using an inner join. But for the sake of clarity, I would personally stick to explicitly using &#8220;inner join&#8221; to reduce any uncertainty or ambiguity when someone is reading/reviewing your query.</p>
